# Component Gallery — Playwright Audit Harness

This is a self-contained Node project that exercises the component
gallery (`examples/component-gallery/`) end-to-end in Chromium and
WebKit. It produces a structured `audit-report.md` listing which
components pass/fail/flake across the four named variants.

## One-time setup

```bash
cd examples/component-gallery/e2e
npm install
npx playwright install --with-deps chromium webkit
```

## Daily commands

| Command | What it does |
|---|---|
| `npm run e2e` | Build the gallery (`dx build --release`) and run the full audit against the static artifact. |
| `npm run e2e:dev-loop` | Run against `dx serve`. You start `dx serve --package component-gallery --port 9173` yourself; the harness emulates the dev experience. |
| `npm run e2e:update` | Same as `e2e` but updates visual snapshots. Review the diff in your PR like any other file change. |
| `npm run e2e:ci` | What CI runs: emits `audit-report.md` and the HTML report. |
| `npm run test:unit` | Vitest unit tests for the helper library and the reporter. |
| `npm run lint:tsc` | Type-check the TS without emitting. |

## Reading `audit-report.md`

Each row groups the worst outcome across the four variants for one
component. Columns:

- **Smoke** — interaction smoke from `tests/smoke.spec.ts`.
- **Motion** — motion contract from `tests/components/*.spec.ts` for
  components whose manifest entry sets `layers.motion: true`.
- **Visual** — pixel snapshot from `tests/visual.spec.ts`.
- **Status** — `ready` if all covered layers pass, `regression` if any
  layer fails, `partial` if any layer is flaky.

The full per-variant breakdown lives in `playwright-report/` (open
`playwright-report/index.html`).

## Re-baselining visuals

```bash
npm run e2e:update
```

WebKit and Chromium baselines are stored separately under
`tests/visual.spec.ts-snapshots/<entry>-<variant>-<browser>-<platform>.png`. CI runs on
ubuntu-24.04; local snapshots from other OSes are not authoritative.
Use `npm run e2e:ci` (or the CI run itself) to refresh baselines for
review.

## Component manifest constraints

The TypeScript manifest at `tests/_lib/component-manifest.ts` MUST
keep one entry per line in the `{ name: "...", slug: "...", ... }`
shape — the Cargo parity test at
`examples/component-gallery/tests/manifest.rs` parses by line. A
Prettier reformat that wraps an entry across multiple lines silently
drops it from the parity check.

## Spring reference fixture

`tests/_lib/spring-reference.json` is generated by the Cargo test
`examples/component-gallery/tests/spring_reference.rs`. To regenerate
after changing `Spring::snappy()` (or adding new presets):

```bash
KINETICS_UPDATE_SPRING_REFERENCE=1 cargo test -p component-gallery --test spring_reference
```

Commit the resulting JSON alongside the Rust change.

## Adding a new component to the audit

1. Add the entry to `examples/component-gallery/src/docs.rs`.
2. Add a matching row to `tests/_lib/component-manifest.ts` with
   accurate `layers` flags (single-line shape; see above).
3. Run `cargo test -p component-gallery --test manifest` to confirm
   parity.
4. Add a bespoke spec under `tests/components/<slug>.spec.ts` (use
   the existing specs as templates by category).
5. If the component has motion: run `KINETICS_UPDATE_SPRING_REFERENCE=1 cargo test -p component-gallery --test spring_reference` if it adds a new spring preset.
6. `npm run e2e` to confirm it passes; commit.

## Known limitations

- **WebKit on Windows / Linux**: Playwright's bundled WebKit is a useful canary
  but is not pixel-identical to ship Safari. Treat WebKit baselines as
  approximate. WebKit may also struggle with the gallery preference-bar
  radio-button click sequence; some variants may not generate baselines
  cleanly on WebKit and need a separate triage pass.
- **`wasm-opt` crash on Windows**: Dioxus 0.7's bundled CLI sometimes fails
  `wasm-opt` with `STATUS_STACK_BUFFER_OVERRUN` and falls back to
  un-optimised WASM. This is harmless for the audit (~5MB WASM still loads
  fine), but means local builds run slower than CI on Linux.
- **WebGPU/WebGL in headless Chromium**: not available; the glass engine
  emits `console.warn` and gracefully degrades to the CSS fallback path.
  The `console-guard` helper tolerates `warn` by design — animation
  regressions are caught by the motion + visual layers, not the smoke layer.
- **dev-loop project**: requires a separately-running `dx serve`. With no
  server up, dev-loop tests fail with `ERR_CONNECTION_REFUSED`. This is
  expected; CI only runs `--project=static` / `--project=static-webkit`.
- **Baselines are platform-keyed.** Playwright suffixes snapshots with
  `process.platform` (e.g. `-win32`, `-linux`). The current branch ships
  only the Windows-local baselines (`*-static-win32.png`,
  `*-static-webkit-win32.png`). The first CI run on `ubuntu-24.04` will
  auto-generate `-linux` baselines on its first invocation (Playwright
  writes the actual screenshot when no baseline exists). Commit those
  back via a follow-up PR for deterministic comparison on subsequent
  CI runs. Local Windows runs continue to compare against the
  committed `-win32` baselines.

## Spec 2 hand-off

The deliverable that connects this Spec 1 harness to Spec 2 (advanced
motion + GPU acceleration) is `audit-report.md`, written next to this
README by the custom Playwright reporter on every run. The file is
**not** git-ignored — commit a freshly-generated report alongside the
merge of this branch so Spec 2's author has a stable, reviewable
catalog of which components currently pass each audit layer. CI also
uploads it as a workflow artifact for any subsequent run.

Known audit signals to triage in Spec 2 (from the current local run):
- `Sequence`, `TimelineScope`, `FrameStage`, `KineticBox` motion specs
  surface DOM-selector mismatches and missing per-frame inline-style
  progression. Each needs case-by-case triage: test-expectation bug
  (spec writer overstated the contract) vs. component bug (motion
  engine does not progress the styles the gallery preview promises).
- `Presence` clock-driven enter/exit passes on Chromium but flakes on
  WebKit, where a sibling tile intercepts pointer events over the
  Replay button. WebKit-specific layout issue worth a CSS pass.
- WebKit preference-bar radio clicks do not register on non-default
  variants; the `mountGallery` helper times out. Either fix
  `controls.rs` to expose a more WebKit-compatible role tree, or
  re-implement `selectRadio` to bypass the click and dispatch a
  synthesized `change` event.
